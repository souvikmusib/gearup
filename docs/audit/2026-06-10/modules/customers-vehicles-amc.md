# Customers, Vehicles, AMC — module audit

_Module key:_ `customers-vehicles-amc`

## Summary

The module is functional but has several go-live concerns. The biggest risk is a race-condition in AMC contract numbering (count+1 will collide under concurrent creates), an AMC double-spend race on `Use Service` (no row-lock; servicesRemaining can go negative under concurrency), and customer/vehicle DELETE that does its existence checks outside the transaction (TOCTOU — a vehicle/job-card can be inserted between the count and the delete). Several mutating routes (vehicle PATCH, plan POST/PATCH, contract PATCH, contract POST/DELETE, usage DELETE) do not call `logActivity`, undermining the audit trail. Schema lacks a unique constraint on `Vehicle.registrationNumber` and `(customerId, vehicleId, status=ACTIVE)` on AmcContract — duplicates can be created. Validation is loose (`email: z.string().optional()` is later transformed but allows weird states; `z.string()` accepts empty `customerId/vehicleId/amcPlanId` only via `.min(1)` on AMC but not on vehicle.customerId; `startDate` is just `z.string()` with no date validation). Every Prisma write uses `data: body as any`, exposing mass-assignment risk if Zod schemas drift. No rate limiting on admin routes (acceptable since auth-gated). UI uses `alert()`/`confirm()` and lacks optimistic rollback on failures; the AMC "Use Service" modal asks the user to paste a job card ID by hand — major UX bug. AMC plan POST has no `logActivity` and no rate limit, but more critically the plan's `coveredItems: z.any()` accepts arbitrary JSON unsanitized.

## Coupling

Customers/Vehicles are referenced by ServiceRequest, Appointment, JobCard, Invoice, AmcContract. Vehicle.customerId is `onDelete: Cascade` in schema — surprising and contradicts the manual existence-check in DELETE. AmcContract → AmcServiceUsage links to JobCard (job-cards module dependency). AMC usage POST is tightly coupled with job-cards flow but does NOT verify the jobCardId belongs to the same customer/vehicle as the contract. ActivityLog read by /customers/[id]/history but never written for vehicle PATCH/DELETE, plan POST/PATCH/DELETE, contract PATCH/POST/DELETE, usage DELETE — audit trail incomplete. Uses shared `paginate`, `handleApiError`, `requirePermission`, `logActivity`.

## Findings

### [P0 · BLOCKER] AmcContract number generated by count()+1 — colliding under concurrency
_id:_ `amc-contract-number-race` · _category:_ race-condition · _file:_ `apps/web/src/app/api/admin/amc/contracts/route.ts:55-56`

```
const count = await prisma.amcContract.count();
const contractNumber = `AMC-${String(count + 1).padStart(5, '0')}`;
```
**Impact.** Two concurrent POSTs both read the same count, both produce the same contract number; one will hit P2002 (unique on contractNumber) and 500/CONFLICT to the user, or worse, contracts created out-of-order leave permanent gaps/dupes if uniqueness is later relaxed.

**Fix.** Use a Postgres sequence / `nextval` via $queryRaw, or wrap count+create inside a serializable $transaction with retry on P2002, or use a dedicated NumberSequence table with `update` returning row.

  _Adversarial verify:_ **CONFIRMED** (now P1) — Confirmed: route.ts:55-56 does prisma.amcContract.count() then create() with no transaction, and contractNumber is @unique in schema.prisma:814. Two concurrent POSTs will produce the same number; one will hit P2002 and 500 to the user. However, this is an admin-only endpoint (requirePermission AMC_CONTRACTS_MANAGE) and the unique constraint prevents actual data corruption — the failure mode is a confusing 500 on a low-frequency manual action, not silent dupes. Real bug worth fixing (sequence or NumberSequence row-lock), but P1 not P0 go-live blocker.

### [P0 · BLOCKER] AMC Use Service has read-then-write race — services can go negative
_id:_ `amc-use-service-no-row-lock` · _category:_ race-condition · _file:_ `apps/web/src/app/api/admin/amc/contracts/[id]/route.ts:57-77`

```
const contract = await tx.amcContract.findUniqueOrThrow({ where: { id: params.id } });
if (contract.servicesRemaining <= 0) throw new ValidationError('No services remaining');
...
await tx.amcContract.update({ where: { id: params.id }, data: { servicesUsed: { increment: 1 }, servicesRemaining: { decrement: 1 } } });
```
**Impact.** Default Prisma tx isolation is READ COMMITTED. Two parallel POSTs read servicesRemaining=1, both pass guard, both decrement → servicesRemaining=-1 and an extra free service is granted. Same race lets an EXPIRED/CANCELLED contract be used if status flips mid-tx.

**Fix.** Use conditional update with WHERE-guard: `updateMany({ where:{ id, status:'ACTIVE', servicesRemaining:{gt:0}, endDate:{gte: now} }, data:{ servicesUsed:{increment:1}, servicesRemaining:{decrement:1} } })` and assert count===1 before inserting usage. Or `SELECT ... FOR UPDATE` via $queryRaw, or set tx isolation to Serializable.

  _Adversarial verify:_ **CONFIRMED** (now P1) — Confirmed at apps/web/src/app/api/admin/amc/contracts/[id]/route.ts:57-80. The POST handler reads the contract inside a default Prisma transaction (READ COMMITTED), checks servicesRemaining/status/endDate in JS, then issues an unconditional increment/decrement update. Two concurrent POSTs can both observe servicesRemaining=1 and both decrement to -1; same window allows a status flip mid-tx to be ignored. No unique constraint, row lock, or conditional WHERE guards this. Downgraded from P0 to P1 because the route requires AMC_CONTRACTS_MANAGE permission (staff-only), so exploitation requires an authenticated insider double-click / parallel request rather than a public attacker, but the data corruption (negative remaining count, granted free service) is genuine and the proposed updateMany-with-WHERE-guard fix is the right remediation.

### [P1] No constraint preventing multiple ACTIVE AMC contracts on same vehicle
_id:_ `amc-contract-duplicate-active` · _category:_ business-logic · _file:_ `apps/web/src/app/api/admin/amc/contracts/route.ts:44-74`

```
export async function POST(req: NextRequest) {
  ...
  const contract = await prisma.amcContract.create({ data: { ... } });
```
**Impact.** User can create overlapping AMC contracts for the same vehicle (intentional or accidental), inflating perceived coverage and revenue accounting; `Use Service` becomes ambiguous which contract to draw from.

**Fix.** Add precheck: `await prisma.amcContract.findFirst({ where:{ vehicleId, status:'ACTIVE', endDate:{gte: now} } })` and reject; add partial unique index `@@unique([vehicleId, status])` (Postgres partial index via raw migration) for ACTIVE only.

### [P1 · BLOCKER] Customer DELETE checks counts outside transaction (TOCTOU)
_id:_ `customer-delete-toctou` · _category:_ race-condition · _file:_ `apps/web/src/app/api/admin/customers/[id]/route.ts:38-48`

```
const [vehicles, jobCards, invoices] = await Promise.all([prisma.vehicle.count(...), prisma.jobCard.count(...), prisma.invoice.count(...)]);
if (vehicles > 0 || jobCards > 0 || invoices > 0) { ... }
await prisma.serviceRequest.deleteMany(...);
await prisma.appointment.deleteMany(...);
await prisma.customer.delete({ where: { id: params.id } });
```
**Impact.** Concurrent POST /vehicles or /job-cards between the count and delete will succeed silently; orphaned children OR cascade-delete loses real records. Worse: schema has `Vehicle.customer ... onDelete: Cascade` — once customer is deleted, ALL vehicles disappear despite the guard.

**Fix.** Wrap entire block in `prisma.$transaction([...])` (interactive tx) and re-count inside the tx; remove `onDelete: Cascade` from Vehicle or make it explicit. Also include amcContracts in the precheck (missing).

  _Adversarial verify:_ **CONFIRMED** (now P1) — Verified in apps/web/src/app/api/admin/customers/[id]/route.ts:35-52. Three count() calls run outside any transaction, then the customer is deleted via separate awaits. Schema confirms Vehicle has `onDelete: Cascade` on the customer relation (schema.prisma:269), so any vehicle inserted via concurrent POST between the count and delete is silently destroyed — the 409 guard provides no real protection. Job cards/invoices use default Restrict, so concurrent inserts of those would instead throw a foreign-key error mid-delete, leaving orphaned serviceRequest/appointment deletions already committed. Also confirmed amcContracts (line 274) is in the precheck list but missing from the count guard. Fix requires wrapping in prisma.$transaction with re-count inside.

### [P2] All Prisma writes use `data: body as any` — mass-assignment / type-safety hole
_id:_ `mass-assignment-as-any` · _category:_ type-safety · _file:_ `apps/web/src/app/api/admin/customers/route.ts:38, /customers/[id]/route.ts:29, /vehicles/route.ts:42, /vehicles/[id]/route.ts:21, /amc/plans/route.ts:35, /amc/plans/[id]/route.ts:33, /amc/contracts/[id]/route.ts:31`

```

...

```
**Impact.** If Zod schema drifts (new field added) and forgotten in cast, Prisma silently accepts unintended fields. Also kills compile-time safety against schema changes — go-live regressions go undetected.

**Fix.** Remove `as any`; let Prisma typecheck. If Zod->Prisma types mismatch (e.g. dates), normalize explicitly in a typed object passed to Prisma.

### [P3] Customer email validator pipes optional through email() — accepts non-canonical empty
_id:_ `customer-email-validator-quirk` · _category:_ validation · _file:_ `apps/web/src/app/api/admin/customers/route.ts:12, /customers/[id]/route.ts:11`

```
email: z.string().optional().transform(v => v?.trim() || undefined).pipe(z.string().email().optional()),
```
**Impact.** `undefined` is allowed at both ends but `null` from client crashes (z.string() rejects null with 'Expected string, received null'). Edit page sends `email || ''`, which trims to '' → transform → undefined → fine, but the UX 'clear email' workflow is fragile.

**Fix.** Use `z.preprocess((v) => (typeof v === 'string' ? v.trim() || undefined : v ?? undefined), z.string().email().optional())`. Same fix for phoneNumber/postalCode if optional clearing wanted.

### [P2] Usage DELETE increments servicesRemaining without status check (overflow risk)
