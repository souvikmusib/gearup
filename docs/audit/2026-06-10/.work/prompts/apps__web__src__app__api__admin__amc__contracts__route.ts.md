### 1. [P0 · BLOCKER] AmcContract number generated by count()+1 — colliding under concurrency
- _id_: `amc-contract-number-race` · _category_: race-condition
- _location_: `apps/web/src/app/api/admin/amc/contracts/route.ts:55-56`
- _evidence_:
```
const count = await prisma.amcContract.count();
const contractNumber = `AMC-${String(count + 1).padStart(5, '0')}`;
```
- _impact_: Two concurrent POSTs both read the same count, both produce the same contract number; one will hit P2002 (unique on contractNumber) and 500/CONFLICT to the user, or worse, contracts created out-of-order leave permanent gaps/dupes if uniqueness is later relaxed.
- _proposed fix_: Use a Postgres sequence / `nextval` via $queryRaw, or wrap count+create inside a serializable $transaction with retry on P2002, or use a dedicated NumberSequence table with `update` returning row.
- _verifier said_: real=True, Confirmed: route.ts:55-56 does prisma.amcContract.count() then create() with no transaction, and contractNumber is @unique in schema.prisma:814. Two concurrent POSTs will produce the same number; one will hit P2002 and 500 to the user. However, this is an admin-only endpoint (requirePermission AMC_CONTRACTS_MANAGE) and the unique constraint prevents actual data corruption — the failure mode is a confusing 500 on a low-frequency manual action, not silent dupes. Real bug worth fixing (sequence or NumberSequence row-lock), but P1 not P0 go-live blocker.

- _evidence_:
